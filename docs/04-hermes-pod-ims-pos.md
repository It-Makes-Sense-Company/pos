## 1. Решение и его контекст

В плане 02 ядро значилось как «зона Султана, отдельная кодовая база на OpenClaw». Развилка 22.06: строим **свою версию POS здесь**, обкатываем, потом сводим с Султаном в **один репозиторий на GitHub-организации IMS** (её надо завести). При этом за основу берём **Hermes Agent** — open-source агент от Nous Research, который Никита назвал референсом.

**Почему это не конфликт с Султаном, а схождение.** Hermes — прямой наследник OpenClaw той же Nous Research. В нём встроена миграция: `hermes claw migrate` (`--dry-run`, `--preset user-data`, `--overwrite`) импортирует из `~/.openclaw` настройки, память (`MEMORY.md`/`USER.md`), пользовательские skills (`→ ~/.hermes/skills/openclaw-imports/`) и ключи. То есть прототип Султана на OpenClaw (обкатанный на Артёме) **переезжает в Hermes**, а не выбрасывается. Сводя версии, мы не выбираем «его или наше» — обе линии впадают в Hermes.

**Лицензия** — MIT, коммерческое использование разрешено.

## 2. Что Hermes закрывает из плана 02 (карта соответствий)

| Составляющая POS (план 02) | Что даёт Hermes из коробки |
|---|---|
| **(E) Бот в Telegram, мультимодальный ввод** | `gateway/` поверх `python-telegram-bot 22.6`. Единый процесс на Telegram/Discord/Slack/WhatsApp/Signal/CLI. По умолчанию long-polling (без вебхука и SSL). Транскрипция голосовых, кросс-платформенная непрерывность диалога. |
| **(B) Файловая память «по Карпатому»** | Состояние — файлами в профиле: `MEMORY.md` / `USER.md` / `SOUL.md` + `skills/`. Плюс SQLite state-DB с **FTS5-поиском по сессиям** и LLM-суммаризацией для cross-session recall. |
| **(D) Sense Engine / инструменты** | 40+ tools + MCP + система **skills** (процедурная память; агент создаёт и сам улучшает навыки). `curator` фоном поддерживает коллекцию (архив/консолидация/пин). Ресёрч — exa / firecrawl; картинки — fal; TTS — edge/eleven. |
| **(E) Проактивные пинги / онбординг** | `cron/` — расписания и «periodic nudges». Два режима: in-process 60-сек тикер (для всегда-живого инстанса — наш случай) или Chronos managed-cron (scale-to-zero для serverless). |
| **(F) Дашборд (зона Насти)** | Готовый веб-дашборд `web/` (React): ProfilesPage / ModelsPage / SkillsPage / McpPage + REST API (`/api/profiles`, `/api/skills`, `/api/mcp/...`). Фундамент для (F), а не верстать с нуля. |
| **(A) Приземление данных / создание профиля** | `docs/design/profile-builder.md` — дизайн dashboard-native создания профиля. Профиль = HERMES_HOME-директория, всё остальное реюзается. |
| Инференс — **OpenRouter** | Провайдер первого класса (`OPENROUTER_API_KEY`, `base_url=https://openrouter.ai/api/v1`). 200+ моделей, переключение без кода (`hermes model`). Поддержка ещё ~20 провайдеров — лок-ина нет. |

Вывод: «скелет + мультимодальный ввод» (первый кусок MVP по плану 02) мы **не пишем — конфигурируем**. Реальная работа смещается на продуктовый слой: схему памяти под персональный бренд, артефакт коммуникационной стратегии (C), логику Sense Engine как skills, онбординг.

## 3. Мультитенантность — закрыта через профили

Ключевой риск был: наш продукт нужен на N человек (команда → клиенты), а Hermes пахнет «персональным агентом на одного». Разбор показал — решается штатно.

- **Профиль = полный изолированный HERMES_HOME:** `~/.hermes/profiles/<name>/` со своими `config.yaml` (модель/провайдер, MCP, включённые skills), `skills/`, `.env` (секреты), `SOUL.md` / `USER.md` (идентичность). Память и персона на профиль изолированы **по построению** — менять модель данных не нужно.
- **Переключение:** флаг `-p <profile>` + context-local `set_hermes_home_override()`. Один процесс может обслуживать несколько профилей, маршрутизируя по входящему пользователю.
- **Доступ:** вместо статичных allowlist'ов — `gateway/pairing.py`: неизвестный пользователь получает одноразовый код, владелец одобряет его через CLI (8-симв. коды, истечение за час, рейт-лимит, лок-аут). Плюс `authz_mixin` — политика, кому вообще можно писать агенту.

**Засада (из profile-builder.md, seam #1):** `tools/skills_hub.py` биндит `SKILLS_DIR = HERMES_HOME/"skills"` на import-time, поэтому установка skills из хаба в чужой профиль идёт **только через subprocess** (`-p <profile>`), а не через runtime-override. Учитываем при автоматизации онбординга.

**Решение по хостингу (22.06): на старте — один личный инстанс на Никиту** (как Hermes и задуман). Мультипрофильность/мультиюзер — следующий шаг, когда обкатаем одиночный. Профили дают чёткий путь туда без переписывания.

## 4. Модель памяти Hermes vs наша (B) «срок давности»

Что есть:
- **`MEMORY.md` / `USER.md` / `SOUL.md`** — персистентные файлы идентичности и памяти (ровно файловый Карпатый-стиль из плана 02).
- **Сессии** — SQLite state-DB, FTS5-поиск + LLM-суммаризация для recall между сессиями.
- **Skills** — процедурная память; `curator` (инактивити-триггер, не демон) проводит lifecycle: архивирует/консолидирует/пинит. **Никогда не удаляет — только архивирует** (восстановимо).
- **Honcho** (`plugins/memory/honcho/`, опциональная зависимость `honcho-ai`) — диалектическое моделирование пользователя («углубляющаяся модель того, кто ты»).
- **Insights** (`agent/insights.py`) — аналитика по state-DB: токены, стоимость, паттерны использования, разбивка по моделям/платформам. Полезно и для контроля бюджета, и как источник метрик для дашборда (F).

**Гэп под нашу задачу:** наша фишка (B) — у каждого факта **вес/срок давности** (как в рекомендалках: 7-летний инфоповод не тащим в сегодняшний пост). У Hermes recency реализован косвенно — через архивацию curator'ом и суммаризацию сессий, **не как явный вес факта**. Это наш кастомный слой: либо skill, который проставляет/учитывает «свежесть», либо конвенция в структуре `MEMORY.md`. Открытый вопрос (см. раздел 7).

## 5. Инференс и стоимость

- **Провайдер:** OpenRouter (как в Pocket Listings — единый ключ, роутинг моделей под задачу). Дешёвая модель для фильтра/классификации (Sense Engine), большая — для склейки профиля (шаг A). Дефолтная модель задаётся в `config.yaml` (`model.default`), не в `.env`.
- **Ключ:** для MVP — общий IMS-ключ OpenRouter. Вопрос «токен клиента vs подписка» откладываем до монетизации (открытый вопрос плана 02 #3).
- **Бюджет:** `agent/insights.py` даёт оценку стоимости из коробки — мониторим с первого дня.

### Ограничения хостинга на SkyStark (важно)
Сервер забит: ~1.2 Gi свободной RAM, swap уже задействован, 8+ сервисов (почта — боевая, ронять нельзя), GPU нет, 4 CPU. Следствия:
- **Локальный инференс невозможен** — только удалённый (OpenRouter). Это и так наш выбор.
- **Локальную транскрипцию не поднимаем.** В core-зависимостях Hermes есть `faster-whisper + numpy + sounddevice` — тяжело для нашего ящика. Голосовые гнать на удалённый ASR (Groq Whisper / OpenAI). Проверить, как Hermes конфигурирует remote-транскрипцию, или вынести в свой tool. (Termux-сборка ставит `.[termux]` без voice-зависимостей — есть путь без них.)
- Сам gateway-процесс лёгкий (инференс удалённый) — один инстанс впишется, но за RAM следить.

## 6. Конфигурация инстанса (как поднимать)

Структура: `~/.hermes/` — `config.yaml` (`get_config_path`), `.env` (`get_env_path`), `skills/`, `sessions` (state-DB), `pairing/`, `plans/`, профили в `profiles/<name>/`.

Минимальный конфиг под IMS POS:
- `.env`: `OPENROUTER_API_KEY=…`, `TELEGRAM_BOT_TOKEN=…` (новый бот через @BotFather под IMS), `TELEGRAM_ALLOWED_USERS=<твой user_id>` (или pairing-код), опц. `TELEGRAM_HOME_CHANNEL` (куда доставлять cron-пинги).
- `config.yaml`: `model.provider: openrouter`, `model.base_url: https://openrouter.ai/api/v1`, `model.default: <модель>`.
- Режим Telegram — long-polling по умолчанию (вебхук/SSL не нужны; `TELEGRAM_WEBHOOK_URL` только если захотим webhook).
- Установка: `uv` через `setup-hermes.sh` (Python 3.11 venv), extra `.[messaging]` для Telegram. На сервере — systemd-сервис (по образцу `pocket-listings.service`).

> Клон для оценки лежит в `/tmp/hermes-eval` (176 MB, `--depth 1`). Для боевого инстанса — чистая установка в место под наш контроль (см. раздел 8).

## 7. Открытые вопросы (обновляют раздел 7 плана 02)

1. **«Срок давности» фактов** — как наложить наш вес/recency-слой на память Hermes (skill vs конвенция в `MEMORY.md` vs Honcho). → раздел 4.
2. **Remote-транскрипция** — встроенный путь Hermes для удалённого ASR или свой tool (Groq/OpenAI Whisper). Не поднимать faster-whisper на SkyStark.
3. **Коммуникационная стратегия (C) в памяти** — как разложить (не монолитный документ): `SOUL.md` + набор skills по пилларам vs отдельные файлы. Стык с (B).
4. **Дашборд (F) Насти** — берём готовый `web/` Hermes как базу или свой на сайте, и как стыкуем с профилями. Метрики — главный открытый вопрос Насти, `agent/insights.py` как источник.
5. **Мультипрофильность** — когда и как переходим от 1 инстанса к профилю-на-пользователя (один процесс с `-p` vs процесс-на-юзера); связано с RAM SkyStark и хостингом клиентов (serverless Modal/Daytona для scale-to-zero).
6. **GitHub-организация IMS** — завести, решить имя, перенести форк/деплой Hermes туда (нужен авторизованный `gh`).
